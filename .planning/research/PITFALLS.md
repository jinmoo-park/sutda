# 도메인 함정 (Domain Pitfalls)

**도메인:** 실시간 멀티플레이어 웹 카드 게임 (온라인 섯다)
**조사일:** 2026-03-29
**신뢰도:** MEDIUM (웹 검색 불가로 훈련 데이터 기반, 실시간 멀티플레이어 게임 개발의 잘 알려진 문제들)

---

## Critical Pitfalls (치명적 함정)

프로젝트를 재작성하게 만들거나 심각한 장애를 유발하는 실수들.

---

### Pitfall 1: 클라이언트 신뢰 기반 게임 상태 관리

**무엇이 잘못되는가:** 클라이언트가 "내 패는 이것이다", "베팅 금액은 이것이다"를 서버에 보내고 서버가 그대로 수용하는 구조. 카드 정보가 클라이언트 메모리나 WebSocket 메시지에 노출되면 브라우저 DevTools로 상대 패를 볼 수 있다.

**왜 발생하는가:** "친구들끼리 하는 게임인데 치팅이 있겠어?"라는 안일한 가정. 초기에 빠른 프로토타이핑을 위해 클라이언트에서 게임 로직을 돌리고, 나중에 서버로 옮기려다 구조 전체를 재작성하게 된다.

**결과:**
- 상대방 패 열람 가능 (DevTools에서 WebSocket 메시지 확인)
- 베팅 금액 조작 가능
- 족보 판정 조작 가능
- "친구끼리" 게임이라도 치팅이 발견되면 신뢰 붕괴, 게임 사용 중단

**예방:**
- **Day 1부터 서버 권위(server-authoritative) 아키텍처 적용**. 서버만 전체 덱과 각 플레이어 패를 알고, 클라이언트에게는 "본인 패"와 "공개 정보"만 전송
- 클라이언트는 액션(콜/레이즈/다이)만 서버에 전송, 서버가 유효성 검증 후 상태 변경
- WebSocket 메시지에 다른 플레이어의 비공개 카드 정보를 절대 포함하지 않음
- 인디언 섯다의 경우 특히 주의: "나만 못 보는 카드"를 클라이언트에 보내면 안 됨

**감지 징후:** 클라이언트 코드에 `deck`, `allCards`, `otherPlayerHand` 같은 변수가 있으면 위험 신호

**관련 페이즈:** Phase 1 (핵심 인프라 구축 시 반드시 서버 권위 구조로 시작)

### Pitfall 2: WebSocket 연결 끊김/재접속 미처리

**무엇이 잘못되는가:** 플레이어의 네트워크가 순간적으로 끊기거나 브라우저 탭이 백그라운드로 가면 WebSocket 연결이 끊어진다. 재접속 후 게임 상태를 복구하지 못해 해당 플레이어가 게임에서 이탈하거나, 나머지 플레이어들이 무한 대기에 빠진다.

**왜 발생하는가:** 개발 환경에서는 연결이 안정적이라 이 문제를 간과함. 실제 모바일 브라우저에서는 화면 전환, 전화 수신, WiFi-셀룰러 전환 등으로 연결이 자주 끊김.

**결과:**
- 재접속한 플레이어가 빈 화면을 보거나 이전 상태를 봄
- 차례인 플레이어가 이탈하면 나머지 전원이 영원히 대기
- 게임 중간에 나간 플레이어의 칩/판돈 처리 불가

**예방:**
- **재접속 시 전체 게임 상태 스냅샷을 전송하는 `rejoin` 핸들러 구현**
- heartbeat/ping-pong으로 연결 상태 감시 (30초 이상 무응답 = 연결 끊김 판정)
- 연결 끊김 시 타이머 시작 → 일정 시간(예: 60초) 내 재접속 없으면 자동 다이 처리
- 클라이언트에 자동 재접속 로직 구현 (exponential backoff)
- 서버에 게임 상태를 항상 메모리에 유지 (WebSocket 연결과 게임 상태를 분리)

**감지 징후:** `socket.on('disconnect')` 핸들러가 없거나, 게임 상태가 소켓 객체에 직접 바인딩되어 있으면 위험

**관련 페이즈:** Phase 1 (WebSocket 인프라 구축 시), Phase 2 (게임 플로우 완성 시 재접속 시나리오 테스트)

### Pitfall 3: 턴 기반 로직의 동시성(Race Condition) 버그

**무엇이 잘못되는가:** 베팅 차례인 플레이어가 "콜"을 누르는 동시에 타임아웃으로 자동 다이가 처리되어 이중 액션이 발생. 또는 두 클라이언트가 거의 동시에 메시지를 보내 서버 상태가 꼬임.

**왜 발생하는가:** WebSocket은 비동기 메시지 기반이라 메시지 도착 순서가 보장되지 않음. 네트워크 지연으로 클라이언트와 서버 간 상태 불일치가 발생.

**결과:**
- 한 플레이어가 두 번 베팅함
- 차례가 아닌 플레이어의 베팅이 처리됨
- 판돈 계산이 맞지 않음
- 구사 재경기 트리거와 일반 승리 판정이 동시에 실행됨

**예방:**
- **서버에서 턴 상태 머신(State Machine) 패턴 적용**: 현재 상태에서 허용되는 액션만 처리, 나머지 무시
- 각 액션에 시퀀스 번호 부여, 기대하지 않은 시퀀스의 메시지 거부
- `currentTurnPlayerId`를 서버에서 관리하고, 해당 플레이어가 아닌 요청은 즉시 거절
- 타이머와 플레이어 액션을 mutex(상호 배제) 패턴으로 처리: 타이머 콜백에서 먼저 "이미 액션 처리됨?" 체크

**감지 징후:** 게임 로직에 `if/else` 분기만 있고 명시적 상태 머신이 없으면 위험. `setTimeout`/`setInterval`과 이벤트 핸들러가 같은 상태를 경쟁적으로 수정하면 위험.

**관련 페이즈:** Phase 1-2 (게임 상태 머신 설계 시 핵심 고려사항)

### Pitfall 4: 5가지 게임 모드를 하나의 거대한 조건문으로 구현

**무엇이 잘못되는가:** 오리지날, 세장, 공유패, 후회의섯다, 인디언을 `if (mode === 'original') ... else if (mode === 'sejang') ...` 식으로 하나의 함수/클래스에 구현. 모드마다 카드 배분 방식, 베팅 라운드 수, 공개/비공개 규칙이 전혀 다른데 이를 하나로 합치면 코드가 스파게티화.

**왜 발생하는가:** 처음에는 "공통 로직이 많으니까 하나로 합치자"라고 생각하지만, 각 모드의 고유 흐름이 복잡해지면서 조건문이 폭발적으로 늘어남.

**결과:**
- 한 모드 수정이 다른 모드를 깨뜨림
- 새 모드 추가 시 전체 코드 이해 필요
- 테스트 불가능한 복잡도
- 세장섯다의 "2라운드 베팅" 같은 고유 흐름이 다른 모드 로직과 충돌

**예방:**
- **공통 인터페이스 + 모드별 구현 분리 (Strategy 패턴)**
  ```
  GameMode (interface)
  ├── OriginalMode    // 2장 → 베팅 → 판정
  ├── SejangMode      // 2장 → 베팅 → 1장 추가 → 선택 → 베팅 → 판정
  ├── SharedCardMode   // 공유카드 선택 → 1장 배분 → 베팅 → 판정
  ├── RegretMode       // 20장 공개 → 드래프트 → 2장 선택 → 베팅 → 판정
  └── IndianMode       // 1장(상대만 봄) → 베팅 → 1장(나만 봄) → 베팅 → 판정
  ```
- 공통 부분만 추출: 족보 판정, 베팅 라운드 실행, 승자 결정, 칩 이동
- 모드별 차이점은 "카드 배분 전략"과 "라운드 구성"으로 분리

**감지 징후:** 하나의 파일이 500줄을 넘으면서 `mode` 변수로 분기하는 코드가 반복되면 위험

**관련 페이즈:** Phase 1 (아키텍처 설계 시), 오리지날 먼저 만들고 나머지 모드를 추가할 때 구조가 확장 가능해야 함

### Pitfall 5: 구사(94) 재경기 로직의 상태 복잡성 과소평가

**무엇이 잘못되는가:** 구사/멍텅구리구사 재경기는 "게임이 끝난 후 조건부로 게임을 다시 시작"하는 메타-게임 로직인데, 이를 일반 게임 플로우에 끼워넣으면 상태 전이가 꼬인다.

**왜 발생하는가:** 재경기는 단순히 "다시 시작"이 아님:
- 판돈이 누적된 상태에서 시작
- 죽었던(다이) 플레이어가 판돈 절반을 내고 재참여 가능
- 재경기 트리거 조건이 다른 플레이어의 패에 의존 (일반 구사: 최고패가 알리 이하, 멍텅구리 구사: 최고패가 팔땡 이하)
- 재경기 참여 의사를 모든 죽은 플레이어에게 물어야 함

**결과:**
- 재경기 시 판돈이 리셋되거나 이중 계산됨
- 재참여한 플레이어의 칩이 제대로 차감되지 않음
- 재경기 조건 판정과 일반 승리 판정이 충돌
- "재경기 중 또 구사가 나오면?" 같은 edge case 미처리

**예방:**
- **재경기를 별도의 게임 상태로 모델링**: `PLAYING` → `ROUND_END` → `GUSA_CHECK` → `GUSA_REJOIN_VOTE` → `GUSA_REPLAY` → `PLAYING`
- 판돈 풀(pot)을 명확하게 분리: 이전 판돈 + 재참여 비용 = 새 판돈
- 재경기 발생 시 "새 라운드"가 아닌 "현재 라운드의 연장"으로 처리
- 재귀적 재경기(재경기 중 또 구사) 처리 로직 명시적 구현: 최대 재경기 횟수 제한 또는 무한 허용 여부 rule_draft에서 확인 필요

**감지 징후:** 게임 상태에 `isRegame` boolean 플래그만 있으면 위험. 재경기 관련 상태가 최소 3개 이상의 단계로 분리되어야 안전.

**관련 페이즈:** Phase 2 (게임 로직 구현 시), rule_draft 기반으로 모든 edge case를 먼저 나열한 후 구현

## Moderate Pitfalls (중간 수준 함정)

개발 속도를 크게 저하시키거나, 사용자 경험을 상당히 해치는 실수들.

---

### Pitfall 6: 카드 정보 시각화와 게임 모드별 공개/비공개 규칙 혼동

**무엇이 잘못되는가:** 각 게임 모드마다 "누가 어떤 카드를 볼 수 있는지"가 완전히 다름. 이 visibility 규칙을 클라이언트에서 관리하면 버그와 치팅 취약점이 동시에 발생.

**구체적 visibility 차이:**
| 모드 | 내 카드 | 상대 카드 | 특수 |
|------|---------|-----------|------|
| 오리지날 | 본인만 봄 | 판정 후 공개 | - |
| 세장 | 본인만 봄 | 판정 후 공개 | 3장 중 2장 선택 |
| 공유패 | 공유 1장 전체 공개 + 개인 1장 | - | 선이 공유패 선택 |
| 후회의섯다 | 20장 전체 공개 → 드래프트 | 드래프트 과정 공개 | 선택 카드는 비공개 |
| 인디언 | 1장은 나만 못 봄, 남들은 봄 | 상대 첫 카드 보임 | 2번째 카드는 나만 봄 |

**예방:**
- 서버가 각 플레이어에게 보내는 메시지에 visibility 필터 적용
- `getVisibleState(playerId, gameMode)` 함수로 플레이어별 맞춤 상태 생성
- 테스트 시 각 모드별로 "플레이어 A가 볼 수 있는 정보" vs "실제 전체 정보"를 비교하는 단위 테스트 작성

**관련 페이즈:** Phase 2-3 (각 게임 모드 구현 시)

---

### Pitfall 7: 베팅 금액 유효성 검증 누락

**무엇이 잘못되는가:** 레이즈 시 "자유 금액"을 입력할 수 있는데, 음수, 0, 보유 칩 초과, 소수점 금액 등 비정상 입력에 대한 서버측 검증이 없음.

**왜 발생하는가:** 클라이언트 UI에서 입력 제한을 걸면 된다고 생각하지만, WebSocket 메시지를 직접 조작하면 우회 가능.

**결과:**
- 칩이 음수가 됨
- 만원 단위가 아닌 금액이 입력됨
- 보유 칩보다 많은 레이즈로 다른 플레이어에게 불공정
- 0원 레이즈로 무한 턴 돌리기

**예방:**
- 서버에서 모든 베팅 액션 검증:
  - `amount > 0`
  - `amount <= player.chips`
  - `amount >= currentBet` (콜 최소 금액)
  - `amount % 10000 === 0` (만원 단위 규칙이 있다면)
- 레이즈 최소 금액 및 최대 금액(올인) 규칙 명확화
- 검증 실패 시 클라이언트에 에러 메시지 반환, 상태 변경하지 않음

**관련 페이즈:** Phase 2 (베팅 시스템 구현 시)

---

### Pitfall 8: 땡값(사이드 페이먼트) 처리의 복잡성

**무엇이 잘못되는가:** 땡값은 "오리지날 모드에서만", "다이한 플레이어 중 땡을 가진 사람이", "최종 승자에게" 지불하는 별도의 금액 흐름인데, 이를 일반 판돈 정산 로직과 섞으면 칩 계산이 틀어진다.

**구체적 복잡성:**
- 다이한 플레이어도 패를 들고 있어야 땡값 판정 가능 → 다이 시 패를 버리면 안 됨
- 땡잡이/암행어사로 이긴 경우 땡값 면제
- 일땡~구땡 = 500원, 광땡/장땡 = 1000원이라는 별도 금액 체계
- 승리한 플레이어의 칩 수령과 판돈 수령이 별도 트랜잭션

**예방:**
- 판돈 정산과 땡값 정산을 **별도 함수**로 분리
- 라운드 종료 시 순서: 1) 승자 결정 → 2) 판돈 정산 → 3) 땡값 계산/정산 → 4) 결과 브로드캐스트
- 다이한 플레이어의 패 정보를 라운드 종료까지 서버에 보관
- 땡값 정산 결과를 UI에 별도로 표시 (판돈과 혼동 방지)

**관련 페이즈:** Phase 2 (오리지날 모드 완성 단계)

---

### Pitfall 9: 방(Room) 수명 주기 관리 부재

**무엇이 잘못되는가:** 방 생성 후 모든 플레이어가 나가거나 연결이 끊겨도 서버 메모리에 방이 남아있음. 또는 방장이 나가면 게임이 중단됨.

**왜 발생하는가:** "친구들끼리 소규모"라서 방 관리를 단순하게 생각하지만, 브라우저 탭 닫기, 새로고침, 장시간 비활성 등 실사용 시나리오가 다양함.

**결과:**
- 메모리 누수로 서버 크래시
- 방장 이탈 시 진행 불가
- 같은 방 코드로 재접속 불가 (방이 이미 사라짐)
- 게임 중간에 나간 플레이어의 칩이 공중에 뜸

**예방:**
- 방 상태 머신: `WAITING` → `PLAYING` → `ROUND_END` → `WAITING` ... → `CLOSED`
- 방장 이탈 시 자동 위임 (다음 플레이어가 방장)
- 모든 플레이어 이탈 후 일정 시간(5분) 유지 → 이후 자동 삭제
- 각 플레이어의 칩 잔액은 방 레벨에서 관리 (연결과 독립)
- 게임 중 이탈 플레이어: 현재 라운드에서 자동 다이 처리, 칩은 그대로 보존

**관련 페이즈:** Phase 1 (방 관리 인프라)

---

### Pitfall 10: 족보 판정 로직의 edge case 누락

**무엇이 잘못되는가:** 족보 판정에서 "광" 여부, "열끗" 여부를 단순히 숫자만으로 판단하면 틀림. 1-1은 일땡이지만, 1광-1은 일땡이고, 3광-8광은 삼팔광땡이지만 3-8(광 없음)은 그냥 일끗.

**구체적 위험 케이스:**
- 카드 1이 두 장인데, 하나는 광이고 하나는 일반 → 일땡이지만 광땡 조합에 사용 가능
- 땡잡이(3+7) vs 삼칠(일반 끗): 3과 7의 조합이 맞지만 족보상 땡잡이는 특수 처리
- 암행어사(열끗4+열끗7) vs 일반 장사(4+10): 카드 속성(열끗 여부)이 핵심
- 구사(4+9) vs 멍텅구리구사(열끗4+열끗9): 열끗 속성에 따라 재경기 조건이 다름

**예방:**
- 카드를 단순 숫자가 아닌 `{ number: number, isGwang: boolean, isYeolKkeut: boolean }` 객체로 모델링
- 족보 판정 함수에 대한 **포괄적 단위 테스트**: 모든 광땡 조합, 모든 땡, 모든 특수 조합, 모든 끗 값 테스트
- 족보 우선순위를 배열/맵으로 명시적 정의 (매직 넘버 사용 금지)
- rule_draft.md의 모든 조합을 테스트 케이스로 변환

**감지 징후:** 카드가 단순 숫자 `number`로만 표현되면 위험

**관련 페이즈:** Phase 2 (족보 시스템 구현, TDD 추천)

## Minor Pitfalls (경미한 함정)

개발자 경험을 저하시키거나 나중에 기술 부채가 되는 실수들.

---

### Pitfall 11: 밤일낮장(선 결정) 시간대 판정 버그

**무엇이 잘못되는가:** 서울 기준 시간을 서버에서 계산할 때 타임존 처리를 잘못하면 낮/밤 판정이 뒤바뀜. 서버가 UTC로 돌아가는데 `new Date().getHours()`를 그대로 쓰면 틀림.

**예방:**
- 서버에서 항상 `Asia/Seoul` 타임존으로 변환 후 시간 판정
- `Intl.DateTimeFormat` 또는 `date-fns-tz` 같은 라이브러리 사용
- 경계값(18:00:00, 06:00:00) 테스트 케이스 작성

**관련 페이즈:** Phase 2 (선 결정 로직)

---

### Pitfall 12: 기리/퉁 애니메이션과 실제 게임 상태 불일치

**무엇이 잘못되는가:** 카드 셔플, 기리(컷), 퉁 연출을 클라이언트 애니메이션으로 구현할 때, 애니메이션 완료와 서버 상태 전이의 타이밍이 맞지 않음.

**예방:**
- 애니메이션은 순수 시각 효과, 게임 상태 전이는 서버 메시지 기반
- 클라이언트가 "기리 완료" 또는 "퉁 선택"을 서버에 전송 → 서버가 배분 결과를 전송 → 클라이언트가 애니메이션 재생
- 애니메이션 스킵 옵션 제공 (빠른 게임 진행 원할 때)

**관련 페이즈:** Phase 3 (UI/UX 구현 시)

---

### Pitfall 13: 후회의섯다 드래프트 UI의 동시 선택 문제

**무엇이 잘못되는가:** 후회의섯다에서 20장이 공개되고 순서대로 1장씩 가져가는데, 동시에 두 플레이어가 같은 카드를 클릭하면 충돌 발생.

**예방:**
- 서버에서 현재 드래프트 차례 플레이어만 선택 가능하도록 제어
- 다른 플레이어의 UI는 "대기 중" 상태로 잠금
- 선택된 카드는 즉시 브로드캐스트하여 다른 클라이언트에서 비활성화

**관련 페이즈:** Phase 3 (후회의섯다 모드 구현 시)

---

### Pitfall 14: WebSocket 메시지 프로토콜 버전 관리 미비

**무엇이 잘못되는가:** 개발 중 메시지 형식을 변경하면 이전 버전 클라이언트와 호환이 깨짐. 배포 후에도 브라우저 캐시 때문에 구 버전 클라이언트가 접속할 수 있음.

**예방:**
- 메시지에 `version` 필드 포함
- 서버에서 클라이언트 버전 확인, 호환 불가 시 새로고침 요청
- TypeScript로 메시지 타입을 서버/클라이언트 간 공유 (monorepo 또는 공유 패키지)

**관련 페이즈:** Phase 1 (프로토콜 설계 시)

---

### Pitfall 15: 칩 재충전 동의 프로세스의 UX 함정

**무엇이 잘못되는가:** 칩이 소진된 플레이어가 재충전을 요청하면 "다른 플레이어의 동의"가 필요한데, 이 투표 프로세스가 게임 흐름을 끊음. 동의/거부 UI가 없으면 구현 자체가 불가.

**예방:**
- 재충전 요청을 게임 라운드 사이에만 허용 (게임 중에는 불가)
- 타이머 설정: 30초 내 응답 없으면 자동 동의/거부 (합의 필요)
- 투표 결과를 모든 플레이어에게 브로드캐스트

**관련 페이즈:** Phase 2 (칩 시스템 완성 시)
