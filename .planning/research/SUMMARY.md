# 프로젝트 리서치 요약

**프로젝트:** 온라인 섯다 (친구들끼리 실시간 멀티플레이어 웹 카드 게임)
**도메인:** 실시간 멀티플레이어 웹 카드 게임
**조사일:** 2026-03-29
**전체 신뢰도:** MEDIUM

## 핵심 요약 (Executive Summary)

온라인 섯다는 WebSocket 기반 실시간 멀티플레이어 카드 게임으로, "서버 권위(Server-Authoritative) 아키텍처"가 절대 원칙이다. 카드 게임의 본질인 정보 은닉(상대 패를 모르는 것)을 보장하려면 모든 게임 로직(셔플, 배분, 족보 판정, 베팅 검증)이 서버에서만 실행되어야 하며, 클라이언트는 자기 패와 공개 정보만 수신하는 "멍청한 터미널" 역할에 국한되어야 한다. 이 원칙을 Day 1부터 적용하지 않으면 나중에 전체 재작성이 불가피하다.

추천 접근법은 TypeScript 모노레포(shared/server/client)로 서버-클라이언트 간 메시지 타입을 공유하고, Socket.IO의 Room 기능으로 방 관리를 구현하는 것이다. React + Vite + Zustand 프론트엔드와 Node.js + Express + Socket.IO 백엔드 조합이 이 규모에 최적이며, DB는 불필요하다(인메모리). 5가지 게임 모드는 Strategy 패턴으로 분리하되, 오리지날 모드를 먼저 완성한 후 나머지를 확장하는 순서가 핵심이다.

가장 큰 위험 요소는 세 가지다. 첫째, 족보 판정 로직의 edge case(광/열끗 속성 구분, 땡잡이/암행어사 특수 처리)로 TDD가 필수이다. 둘째, WebSocket 연결 끊김/재접속 처리가 모바일 브라우저 환경에서 반드시 필요하다. 셋째, 구사(94) 재경기의 상태 복잡성(판돈 누적, 죽은 플레이어 재참여, 재귀적 재경기)을 과소평가하기 쉽다. 이 세 가지를 초기 설계에서 명시적으로 다루면 프로젝트 성공 확률이 크게 높아진다.

## 핵심 발견사항

### 추천 스택

TypeScript 모노레포 구조가 핵심이다. `shared/` 패키지에 카드, 족보, 게임 상태, 메시지 프로토콜 타입을 정의하고 서버/클라이언트가 공유하면, 실시간 게임의 가장 흔한 버그 원인인 메시지 불일치를 컴파일 타임에 차단할 수 있다. DB 없이 인메모리로 운영하는 것은 프로젝트 요구사항(세션 단위, 영구 저장 불필요)에 완벽히 부합한다.

**핵심 기술:**
- **TypeScript (~5.7+):** 서버/클라이언트 공유 타입 -- 메시지 계약을 컴파일 타임에 강제
- **Socket.IO (~4.8+):** Room 기반 실시간 통신 -- 방 관리, 자동 재연결, 이벤트 모델이 카드 게임에 최적
- **React (~19.x) + Vite (~6.x):** 컴포넌트 기반 UI + 빠른 개발 서버 -- 카드 테이블 모듈화에 적합
- **Zustand (~5.x):** 경량 상태 관리 -- Socket 이벤트 핸들러에서 직접 접근 가능
- **Node.js (~22.x LTS) + Express (~5.x):** WebSocket 서버 런타임 -- 이벤트 루프 기반, 소규모에 충분
- **zod (~3.x):** 런타임 메시지 유효성 검증 -- TypeScript 타입만으로는 치팅/잘못된 메시지 차단 불가
- **인메모리 (DB 없음):** 방 세션 종료 시 데이터 소멸 -- 요구사항에 완벽 부합

**배포:** Railway 또는 Render (WebSocket 상시 연결 필요, Vercel/Netlify 서버리스는 부적합)

### 필수 기능 (Features)

**반드시 있어야 하는 기능 (Table Stakes):**
- 실시간 게임 상태 동기화 (WebSocket, 서버 권위 모델)
- 방 생성 + 링크 초대 + 닉네임 입장 (핵심 가치)
- 정확한 족보 판정 (광땡 3종 > 땡 10종 > 특수패 6종 > 끗)
- 베팅 시스템 (콜/레이즈/다이, 자유 금액)
- 칩/포인트 추적 (만원 단위, 재충전)
- 턴 관리 + 타이머 (무한 대기 방지)
- 원형 테이블 UI (2~6인 동적 배치, 와이어프레임)
- 연결 끊김 처리 (자동 재연결 + 자동 다이)
- 5가지 게임 모드 전체 (오리지날/세장/한장공유/후회의섯다/인디언)
- 기리/퉁 + 밤일낮장 선 결정

**차별화 기능 (v1.x에서 추가):**
- 땡값 자동 정산 (오리지날 전용)
- 94재경기 자동 트리거 (구사/멍텅구리구사)
- 게임 내 채팅, 족보 힌트

**v2+로 이연:**
- 사운드 효과, 이모지 반응, 패 공개 연출
- 화투 이미지 에셋 교체 (와이어프레임 -> 실제 이미지)
- 모바일 전용 최적화, 게임 로그

### 아키텍처 접근법

Room 기반 서버 권위(Authoritative Server) 아키텍처를 사용한다. 게임 엔진은 FSM(Finite State Machine)으로 12개 상태(WAITING ~ ROUND_END)를 관리하며, 5가지 게임 모드는 Strategy 패턴으로 분리한다. 각 플레이어에게는 Player View Projection을 통해 "해당 플레이어가 볼 수 있는 정보"만 필터링해서 전송한다.

**핵심 컴포넌트:**
1. **shared/ (공유 타입):** 메시지 프로토콜, 카드 정의, 족보 상수 -- 서버/클라이언트 계약
2. **GameEngine (FSM):** 게임 상태 전이 관리 -- 프로젝트의 두뇌
3. **HandEvaluator:** 족보 판정 순수 함수 -- 게임의 핵심, TDD 필수
4. **CardDeck:** 20장 덱 관리, 셔플, 기리/퉁 -- 순수 함수, 독립 테스트 가능
5. **GameModeStrategy:** 5가지 모드별 카드 분배/라운드 구성 -- Strategy 패턴
6. **RoomManager:** 방 생성/삭제, 플레이어 관리 -- Socket.IO Room 활용
7. **BettingManager:** 베팅 규칙, 판돈 추적, 유효성 검증
8. **WebSocket Gateway:** 연결 관리, 메시지 라우팅, Player View Projection 적용

### 치명적 함정 (Top 5)

1. **클라이언트 신뢰 기반 설계** -- Day 1부터 서버 권위 아키텍처 적용. 클라이언트에 다른 플레이어의 비공개 카드 정보를 절대 전송하지 않음
2. **WebSocket 연결 끊김 미처리** -- rejoin 핸들러 구현, heartbeat 감시, 60초 무응답 시 자동 다이, 게임 상태를 WebSocket 연결과 분리
3. **턴 기반 동시성(Race Condition) 버그** -- FSM으로 현재 상태에서 허용되는 액션만 처리, 타이머와 플레이어 액션 간 상호 배제
4. **5가지 모드를 거대 조건문으로 구현** -- Strategy 패턴으로 모드별 분리, 공통 부분(족보, 베팅, 승자 결정)만 추출
5. **구사(94) 재경기 상태 복잡성** -- 재경기를 별도 게임 상태(GUSA_CHECK -> GUSA_REJOIN_VOTE -> GUSA_REPLAY)로 모델링, 판돈 풀 분리

## 로드맵 시사점

리서치 결과에 기반한 페이즈 구조 제안:

### Phase 1: 프로젝트 기반 + 공유 타입 + 핵심 게임 로직
**근거:** 모든 것의 기반이 되는 shared 타입, 카드 모델, 족보 판정을 먼저 확립해야 한다. 네트워크 없이 순수 함수로 테스트 가능한 영역이므로 가장 빠르게 검증할 수 있다.
**결과물:** 모노레포 구조, shared 패키지(카드/족보/메시지 타입), CardDeck, HandEvaluator (+ 포괄적 단위 테스트)
**해당 기능:** 카드 덱(20장), 족보 판정 엔진
**회피할 함정:** 카드를 단순 숫자로 모델링 (반드시 광/열끗 속성 포함 객체로), 족보 edge case 누락 (TDD)

### Phase 2: WebSocket 인프라 + 방 관리
**근거:** 게임 로직이 준비된 후 네트워크 레이어를 구축한다. Socket.IO Room과 방 코드 기반 입장이 프로젝트의 핵심 가치("링크 하나로 플레이")를 실현한다.
**결과물:** WebSocket Gateway, RoomManager, 방 생성/참여 API, 연결/재연결 처리, 메시지 프로토콜
**해당 기능:** 실시간 동기화, 방 생성 + 링크 초대, 닉네임 입장, 연결 끊김 처리
**회피할 함정:** 클라이언트 신뢰 설계, 재접속 미처리, 방 수명 주기 관리 부재

### Phase 3: 오리지날 모드 전체 플로우 (서버)
**근거:** 오리지날 모드가 나머지 4가지 모드의 기반이다. GameEngine FSM, BettingManager, 턴 관리, 기리/퉁, 밤일낮장을 포함한 완전한 1판 플로우를 서버에서 완성한다.
**결과물:** GameEngine (FSM), BettingManager, OriginalMode Strategy, 턴/타이머, 선 결정, 기리/퉁
**해당 기능:** 오리지날 모드, 베팅, 칩 추적, 턴 관리, 밤일낮장, 기리/퉁
**회피할 함정:** 턴 동시성 버그 (FSM + 상호 배제), 베팅 금액 검증 누락

### Phase 4: 클라이언트 UI (와이어프레임)
**근거:** 서버 로직이 완성된 후 클라이언트를 구축한다. 서버가 먼저 완성되어야 클라이언트가 "멍청한 터미널"로 올바르게 설계된다.
**결과물:** 로비(방 참여) UI, 원형 테이블 레이아웃, 카드 표시(와이어프레임), 베팅 UI, 게임 결과 화면
**해당 기능:** 원형 테이블 UI, 베팅 액션 UI, 게임 결과 화면
**회피할 함정:** 클라이언트에서 게임 로직 실행, 전체 상태 브로드캐스트

### Phase 5: 추가 게임 모드 (4종)
**근거:** 오리지날 기반 위에 Strategy 패턴으로 모드를 하나씩 추가한다. 세장 -> 한장공유 -> 인디언 -> 후회의섯다 순서로 복잡도가 증가하는 순으로 구현한다.
**결과물:** SejangMode, SharedCardMode, IndianMode, RegretMode Strategy + 각 모드별 UI 확장
**해당 기능:** 세장섯다, 한장공유, 인디언섯다, 후회의섯다
**회피할 함정:** 거대 조건문 (Strategy 분리), visibility 규칙 혼동 (모드별 getVisibility 함수), 후회의섯다 드래프트 동시 선택

### Phase 6: 땡값 + 94재경기 + 칩 재충전
**근거:** 핵심 게임 플로우(5가지 모드)가 안정된 후 부가 규칙을 추가한다. 이 기능들은 기본 판정/정산 위에 얹히는 추가 레이어이므로 분리 구현이 자연스럽다.
**결과물:** 땡값 정산 모듈, 구사/멍텅구리구사 재경기 상태 머신, 칩 재충전 투표
**해당 기능:** 땡값 자동 정산, 94재경기, 칩 재충전 동의
**회피할 함정:** 구사 재경기 상태 복잡성 (전용 상태 머신), 땡값과 판돈 정산 혼합 (별도 함수 분리)

### Phase 7: 폴리시 + 배포
**근거:** 모든 게임 로직이 완성된 후 사용자 경험 개선과 배포를 진행한다.
**결과물:** 채팅, 족보 힌트, 기본 애니메이션, Railway/Render 배포, 모바일 반응형 기본 대응
**해당 기능:** 게임 내 채팅, 족보 힌트, 모바일 반응형 UI

### 페이즈 순서 근거

- **의존성 기반:** shared 타입 -> 게임 로직(순수 함수) -> 네트워크 -> 서버 통합 -> 클라이언트 -> 모드 확장 -> 부가 규칙
- **아키텍처 패턴 기반:** Layer 0(shared) -> Layer 1(순수 로직) -> Layer 2(GameEngine) -> Layer 3(네트워크) -> Layer 4(UI) 순서가 ARCHITECTURE.md의 빌드 순서와 일치
- **함정 회피:** 서버를 먼저 완성하면 "클라이언트 신뢰" 함정을 원천 차단. Strategy 패턴을 오리지날 완성 후 적용하면 "거대 조건문" 함정 차단. 재경기를 마지막에 구현하면 기본 플로우 안정성 확보 후 복잡한 상태 추가 가능

### 리서치 플래그

페이즈별 심층 리서치가 필요한 항목:
- **Phase 1:** rule_draft.md의 모든 족보 edge case를 테스트 케이스로 변환 필요 -- 광/열끗 조합이 복잡
- **Phase 5 (인디언섯다):** "나만 못 보는 카드" visibility 로직이 다른 모드와 완전히 다름 -- Player View Projection 설계 검증 필요
- **Phase 5 (후회의섯다):** 20장 공개 드래프트 UI/UX가 독특함 -- 참고할 레퍼런스가 거의 없음
- **Phase 6 (94재경기):** 재귀적 재경기(재경기 중 또 구사), 재참여 판돈 계산 등 edge case가 많음 -- rule_draft 정밀 분석 필요

표준 패턴으로 충분한 항목 (리서치 스킵 가능):
- **Phase 2 (WebSocket 인프라):** Socket.IO Room + 재연결은 잘 문서화된 표준 패턴
- **Phase 4 (클라이언트 UI):** React + Zustand + Socket.IO 통합은 레퍼런스 풍부
- **Phase 7 (배포):** Railway/Render Node.js 배포는 단순 표준 절차

- **Railway/Render 현재 요금/기능:** WebSocket 지원 여부와 무료 티어 제한 사항 배포 전 확인
