# Phase 10.1: Stitch 연계 UI 고도화 - Context

**수집일:** 2026-04-01
**상태:** 플래닝 준비 완료

<domain>
## Phase Boundary

Google Stitch MCP를 통해 전체 주요 화면과 모달의 UI를 프로페셔널하게 개선한다.
워크플로우 파이프라인 확립이 아닌 **실제 UI 개선 결과물** 납품이 목표다.

**대상 화면:**
- MainPage (최초화면 — 닉네임 입력 포함)
- WaitingRoom (초대받은 사람 대기 화면)
- RoomPage / GameTable (게임 메인 화면)
- BettingPanel, HandPanel, InfoPanel, ChatPanel (패널 컴포넌트)
- 각종 모달 전체 (AttendSchool, RechargeVote, ModeSelect, GusaAnnounce, GusaRejoin, Golla, SharedCardSelect, Sejang*, MuckChoice, LeaveRoom, DealerSelect, DealerResult, ReturnFromBreak, Shuffle, Cut)
- 버튼 및 공통 UI 요소 전체

**포함하지 않는 것:** 게임 로직(Zustand store, Socket.IO 핸들러) 변경 없음. Phase 11 범위(채팅 구현, 게임 이력, 올인 POT 등) 없음.

</domain>

<decisions>
## 구현 결정사항

### 워크플로우 방식
- **D-01:** A안 채택 — 기존 컴포넌트 파일을 건드리지 않고, Stitch로 만든 새 껍데기 파일을 병렬로 생성한다. 검증이 끝난 것만 선택적으로 기존 파일과 swap. 실패 시 롤백 비용 없음.
- **D-02:** 완벽한 분리 리팩토링 선행 없음. 기존 컴포넌트에서 prop interface만 추출하여 Stitch에 컨텍스트로 제공.

### 대상 컴포넌트 범위
- **D-03:** 전체 주요 화면 및 모달 커버. 단계별 진행:
  1. MainPage (최초 진입점, 첫인상)
  2. WaitingRoom (방 참여 흐름)
  3. GameTable + PlayerSeat (게임 핵심 화면)
  4. BettingPanel + HandPanel + InfoPanel (패널류)
  5. 모달 전체 (우선순위: 자주 쓰이는 것부터)
  6. 공통 버튼/badge/card 등 shadcn/ui 커스터마이징
- **D-04:** 각 컴포넌트는 독립적으로 처리. 완료된 것부터 즉시 swap.

### DESIGN.md 구성 수준
- **D-05:** 스택 + 디자인 토큰 + prop interfaces 포함. 구체적으로:
  - 기술 스택: React + TypeScript + Tailwind CSS + shadcn/ui
  - 현재 색상/타이포그래피/spacing 토큰 (tailwind.config에서 추출)
  - 각 대상 컴포넌트의 TypeScript prop interface
  - 레이아웃 패턴: 3열 데스크탑 / 수직 모바일 (Phase 10 D-16/D-17 결정 유지)
  - 섯다 게임 특성: 화투 카드, 베팅 액션, 한국어 UI
- **D-06:** DESIGN.md를 Stitch에 넘기기 전 Claude Code가 먼저 작성. Stitch가 그 제약 안에서 UI를 생성.

### 검증 기준
- **D-07:** 진무가 개발 서버에서 눈으로 직접 확인 후 swap 여부 결정. 자동화 검증 없음.
- **D-08:** 기존 테스트 통과는 swap 후 확인 (리그레션 방지용 — 통과가 swap 전제 조건은 아님).

### 디자인 컨셉
- **D-09:** 전체 UI 무드 — "공장에서 군용담요 깔아놓고 하는 느낌". 광택 없는 올리브 드랩/카키 군용 팔레트, 거친 산업적 텍스처, 유틸리타리안 미학. 폴리시된 카지노가 아닌 민간 지하 섯다판 느낌.
  - 컬러: 군용 올리브 드랩(OD green), 카키, 다크 카모, 바랜 군용 그린
  - 텍스처: 담요/캔버스 패브릭 느낌, 거친 표면, 노출 콘크리트/철판
  - 타이포: 굵고 실용적, 명조(KimjungchulMyungjo) 유지
  - 버튼/UI: 군용 장비 느낌, 각진 엣지 허용, 미니멀한 장식

### Claude's Discretion
- Stitch 프롬프트 구성 방식 (Claude가 각 컴포넌트별 프롬프트를 어떻게 구조화할지)
- 컴포넌트별 처리 순서 세부 조정 (진행 중 피드백에 따라 유연하게)
- Tailwind inline style fallback 처리 방법 (Stitch output 정규화 시)

## Existing Code Insights

### 현재 기술 스택
- React 19 + TypeScript + Vite
- Tailwind CSS + shadcn/ui (button, badge, card, dialog, input, separator, sonner)
- Zustand (상태 관리) + Socket.IO (실시간 통신)
- 폰트: 김정철명조 (로컬 적용 완료)

### Reusable Assets
- shadcn/ui 컴포넌트들: 새 껍데기에서도 재사용 가능
- `img/` 디렉토리: 카드 이미지, 배경 이미지 (Phase 10 적용 완료)

### Integration Points
- 각 컴포넌트의 prop interface = Stitch와의 계약서 역할
- Zustand store는 컨테이너(RoomPage, App)에서 연결 — 껍데기 컴포넌트는 props만 받음

### 처리 패턴
- 새 컴포넌트: `ComponentName.v2.tsx` 또는 별도 `/stitch/` 디렉토리로 병렬 생성
- 검증 후: 원본 파일 교체 (git으로 롤백 가능)
