## 요약

Phase 10.1은 Google Stitch MCP를 통해 섯다 게임의 전체 UI를 프로페셔널하게 개선하는 작업이다. 핵심 워크플로우는 3단계다: (1) DESIGN.md 작성 (디자인 시스템 + prop interface 계약서), (2) Stitch MCP로 각 컴포넌트의 새 껍데기 생성, (3) 개발 서버에서 육안 확인 후 원본 swap.

Stitch MCP의 도구 파라미터는 공식 문서가 Google 서버 렌더링으로 접근 불가하여 커뮤니티 소스에서 파악했다. 도구 목록은 확인됐으나 `create_design_system`, `apply_design_system`, `edit_screens`, `generate_variants`의 정확한 파라미터 스키마는 실행 시점에 도구 직접 호출로 확인해야 한다.

**핵심 권고사항:** Stitch output은 raw HTML/CSS이므로 React + Tailwind 컴포넌트로 수동 변환이 필요하다. 플래너는 각 컴포넌트마다 "Stitch 생성 → React 변환 → prop 연결 → 개발 서버 확인 → swap" 5단계 태스크를 설계해야 한다.

### 권장 호출 순서

```
1. mcp__stitch__create_project
   → projectName: "섯다-UI"
   → (반환값에서 projectId 추출)

2. mcp__stitch__create_design_system
   → DESIGN.md 내용을 컨텍스트로 제공
   → (반환값에서 designSystemId 추출)

3. mcp__stitch__generate_screen_from_text  ← 컴포넌트별 반복
   → projectId + prompt (컴포넌트 설명 + 제약사항)
   → (반환값에서 screenId 추출)

4. mcp__stitch__apply_design_system
   → designSystemId + screenId
   → 디자인 토큰 일관성 적용

5. mcp__stitch__get_screen
   → screenId
   → HTML/CSS output 추출

6. (필요 시) mcp__stitch__edit_screens
   → screenId + 수정 프롬프트

7. (필요 시) mcp__stitch__generate_variants
   → screenId + variantCount + creativeRange
```

**주의:** `create_design_system`, `apply_design_system`, `edit_screens`, `generate_variants`의 정확한 파라미터 스키마는 공식 문서가 SSR 렌더링으로 접근 불가하여 확인되지 않음. 실행 시점에 도구를 직접 호출하거나 에러 메시지에서 스키마를 파악해야 한다.

### Stitch Output 처리 방식

Stitch는 raw HTML/CSS를 반환한다. React + Tailwind 컴포넌트로 변환 시:

1. HTML 구조 → JSX 변환 (className으로 치환, self-closing 태그)
2. inline style → Tailwind 클래스로 변환 (가능한 경우) 또는 style prop 유지
3. 하드코딩된 텍스트 → props로 치환
4. 이벤트 핸들러 연결 (onClick 등)
5. 기존 prop interface 계약 준수 확인

## 흔한 함정

### 함정 1: Stitch output의 inline style 남용
**발생 원인:** Stitch가 생성하는 HTML은 Tailwind CSS 기반이 아닌 inline style 또는 임의 CSS class 사용.
**방지책:** 변환 시 Tailwind 클래스로 정규화. Tailwind에 없는 값은 `style` prop으로 보존 (특히 게임 특화 색상).
**경고 신호:** `style="color: #xyz"` 패턴 다수 발생 시 정규화 작업 필요.

### 함정 2: prop interface 계약 파기
**발생 원인:** v2 컴포넌트가 원본과 다른 props를 받으면 RoomPage에서 교체 불가.
**방지책:** v2 컴포넌트는 원본 prop interface를 완전히 준수해야 함. 추가 props는 선택적(optional)으로만.
**경고 신호:** `Property 'X' does not exist on type` TypeScript 오류.

### 함정 3: Stitch output에 게임 로직 포함 시도
**발생 원인:** 프롬프트에 게임 상태, 소켓 이벤트 등을 설명하면 Stitch가 불필요한 로직을 생성.
**방지책:** 프롬프트는 시각적 설명만. "콜 버튼이 클릭 시 bet-action 소켓 이벤트를 emit한다" 같은 로직 설명 제외.
**경고 신호:** Stitch output에 `socket`, `store`, `useEffect` 등이 포함된 경우.

### 함정 4: HwatuCard 3D flip 파괴
**발생 원인:** PlayerSeat이나 HandPanel v2에서 HwatuCard 대신 단순 `<img>` 사용.
**방지책:** v2 컴포넌트도 반드시 `HwatuCard` 컴포넌트를 import하여 사용.
**경고 신호:** Phase 10에서 완성된 flip 인터랙션이 동작하지 않는 경우.

### 함정 5: Stitch MCP 도구 파라미터 오류
**발생 원인:** `create_design_system`, `edit_screens` 등의 정확한 파라미터가 문서화되어 있지 않음.
**방지책:** Wave 0에서 `mcp__stitch__list_projects`로 연결 확인 → `create_project` 호출 → 에러 메시지에서 스키마 파악.
**경고 신호:** `Invalid parameter` 또는 `Unknown tool` 오류.
