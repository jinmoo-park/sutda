## 요약

이 페이즈는 실서비스 운영 중인 섯다 게임 서버(Oracle VM, Node.js + Socket.IO + Express)에 대한 보안 감사다. `pnpm audit` 의존성 스캔은 이미 통과했고, 핵심 이슈는 **코드 레벨**과 **인프라 설정**에 집중된다.

발견된 Critical 이슈 2건은 즉시 수정이 필요하다: (1) `.gitignore`에 `.env` 누락 — `ROOM_CREATE_PASSWORD` 등 시크릿이 git에 노출될 위험. (2) CORS fallback이 `true`(모든 origin 허용) — VM에 `CLIENT_ORIGIN` 환경변수 설정 여부 확인 필수. Rate Limiting은 Medium에서 즉시 수정으로 격상됐다 — Oracle VM 1GB RAM 환경에서 이벤트 폭발로 서버 다운 위험이 실재하기 때문이다.

접근 제어(A01) 측면에서 현재 서버는 `socket.data.playerId`를 기반으로 게임 액션을 처리하지만, **일부 이벤트 핸들러에서 요청자가 실제 해당 방의 플레이어인지 검증하지 않는 패턴**이 발견된다 — 이를 체계적으로 감사해야 한다.

**Primary recommendation:** `.env` gitignore 즉시 추가 → CORS 환경변수 VM 확인 → nginx 보안 헤더 추가 → Socket.IO rate limiting 미들웨어 삽입 → 접근 제어 이벤트 핸들러 감사 순서로 진행하라.

## Architecture Patterns

### 현재 서버 보안 현황 (코드 직접 확인)

**`packages/server/src/index.ts` 주요 보안 패턴:**

```
현재 코드 상태 요약:
- CORS: origin: process.env.CLIENT_ORIGIN || true  ← Critical: true가 fallback
- 입력 검증: RoomManager.validateChips() 존재, 닉네임 길이 제한 없음
- 챗 텍스트: text.trim().slice(0, 200) ← 200자 제한 존재
- 에러 코드: emitError() 패턴으로 표준화됨
- 게임 액션: socket.data.playerId로 식별 (set-player-id 패턴)
- 방장 검증: canStartGame()에서 host 여부 확인
- 방 비밀번호: ROOM_CREATE_PASSWORD 환경변수 기반 (Phase 14 구현 완료)
```

**`packages/client/src/main.tsx` 보안 이슈 (Medium):**
```typescript
// 에러 핸들러에서 document.body.innerHTML 직접 삽입 — XSS 가능성
window.onerror = (msg, src, line, col, err) => {
  document.body.innerHTML = `<pre ...>${msg}\n${src}:${line}:${col}\n${err?.stack ?? ''}</pre>`;
};
```
이 코드는 외부 스크립트 주입으로 오염된 `msg`/`err.stack`이 실행될 수 있는 Medium XSS 위험. 그러나 이 에러 핸들러는 브라우저에서 다른 스크립트 오류가 없는 한 실행되지 않고, 게임 서버 context에서 공격 경로가 매우 좁아 Medium으로 분류. D-06에 따라 보고서 기록만.

### Pattern 1: Socket.IO Rate Limiting (in-memory, D-07)

**방식:** 연결 시점 미들웨어 + 이벤트별 소비 패턴

```typescript
// Source: https://github.com/animir/node-rate-limiter-flexible/wiki/Overall-example
import { RateLimiterMemory } from 'rate-limiter-flexible';

// 소켓당 초당 20개 이벤트 (D-07: "소켓당 초당 20개")
const rateLimiter = new RateLimiterMemory({
  points: 20,       // 최대 토큰 수
  duration: 1,      // 1초 window
});

// io.on('connection') 핸들러 내부에서
socket.use(async ([event, ...args], next) => {
  try {
    // socket.id 기반으로 rate limit (IP 공유 환경에서 더 정확)
    await rateLimiter.consume(socket.id);
    next();
  } catch {
    // 한도 초과: 이벤트 무시 (연결 유지)
    next(new Error('RATE_LIMITED'));
  }
});
```

**socket.use()는 Socket.IO v4 공식 지원 패턴이다.** `io.use()` (연결 미들웨어)와 구별 — `socket.use()`는 연결 후 각 이벤트에 실행.

### Pattern 2: nginx limit_conn (연결 수 제한, D-07)

**방식:** IP당 WebSocket 연결 최대 5개 — 자동 스캐너/봇 차단

```nginx
# Source: https://nginx.org/en/docs/http/ngx_http_limit_conn_module.html
# http{} 블록에 추가:
limit_conn_zone $binary_remote_addr zone=sutda_ws_conn:10m;

# server{} 또는 location /socket.io/ 블록에 추가:
location /socket.io/ {
    limit_conn sutda_ws_conn 5;   # IP당 동시 연결 최대 5개
    proxy_pass http://localhost:3001;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
}
```

**주의:** `limit_req_zone` (초당 요청 제한)은 WebSocket 업그레이드 이후 persistent connection에는 적용 안 됨. `limit_conn`이 WebSocket에 유효한 nginx 레이어 방어다.

### Pattern 3: nginx 보안 헤더 (D-05)

```nginx
# Source: https://content-security-policy.com/examples/nginx/
# server{} 블록에 추가 (HTTPS 서버 블록):

# 이미 HSTS는 certbot/nginx Let's Encrypt가 추가함 — 중복 추가 불필요
add_header X-Content-Type-Options "nosniff" always;
add_header X-Frame-Options "SAMEORIGIN" always;

# CSP: WebSocket wss:// 는 connect-src에 명시적으로 지정해야 함
# 'self' 는 ws:// 스킴을 커버하지 않음 (MDN 확인)
add_header Content-Security-Policy "default-src 'self'; connect-src 'self' wss://sutda.duckdns.org; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data:;" always;
```

**CSP 핵심 주의사항:** `connect-src 'self'`만으로는 wss:// WebSocket 연결이 차단된다. `wss://sutda.duckdns.org`를 명시적으로 추가해야 한다.

### Pattern 4: CORS 수정 (D-04)

```typescript
// 현재 (취약):
cors: {
  origin: process.env.CLIENT_ORIGIN || true,  // true = 모든 origin 허용
}

// 수정 후 (안전):
cors: {
  origin: process.env.CLIENT_ORIGIN || 'https://sutda.duckdns.org',  // fallback을 명시적 도메인으로
  methods: ['GET', 'POST'],
}
```

VM에서 `CLIENT_ORIGIN=https://sutda.duckdns.org` 환경변수 설정 여부를 먼저 확인. PM2 ecosystem.config.cjs의 `env` 섹션 또는 `/etc/environment`에서 확인.

### Pattern 5: 접근 제어 감사 체크리스트 (A01)

현재 서버에서 이벤트 핸들러가 다음을 검증해야 한다:

1. **요청자가 해당 방에 속하는가?** — `socket.data.roomId === roomId` 또는 `socket.rooms.has(roomId)` 확인
2. **요청자의 플레이어 ID가 유효한가?** — `socket.data.playerId` 존재 여부
3. **방장 권한 필요 이벤트** — `start-game`, `select-mode` 등은 `canStartGame()` 패턴으로 검증
4. **자기 턴 검증** — `bet-action`, `select-dealer-card`, `attend-school` 등은 GameEngine 내부에서 turn 검증

**현재 코드에서 발견된 패턴:**
- `canStartGame(roomId, socket.id)` — 방장 검증 있음 (OK)
- `getEngine(roomId)` — 엔진 존재 확인 있음 (OK)
- `socket.data.playerId`로 게임 액션 전달 — 서버 권위 모델 준수 (OK)
- `send-chat` 핸들러: `roomId`가 `socket.data.roomId`와 일치하는지 검증 없음 — 잠재적 A01 (Medium)

### Anti-Patterns to Avoid

- **`helmet.js` 추가**: D-05 결정에 따라 Node.js 레이어 헤더 추가 금지. nginx만 사용.
- **Redis 기반 rate limiter**: Oracle VM 1GB RAM에 Redis 추가는 오버엔지니어링. `RateLimiterMemory`로 충분.
- **`limit_req_zone` 사용**: HTTP 요청만 제한 — WebSocket 연결 업그레이드 이후에는 효과 없음.
- **`io.use()` rate limiter**: 연결 미들웨어는 소켓 연결 시 1회만 실행 — 이벤트당 제한에는 `socket.use()` 필요.

## Don't Hand-Roll

| 문제 | 직접 구현 금지 | 사용할 것 | 이유 |
|------|---------------|-----------|------|
| 이벤트별 rate limiting | 커스텀 Map/counter 타이머 | `rate-limiter-flexible` (RateLimiterMemory) | 토큰 버킷 알고리즘, 메모리 누수 방지, 레이스 컨디션 처리 내장 |
| IP 연결 수 제한 | 커스텀 nginx 모듈 | `limit_conn_zone` + `limit_conn` | nginx 내장 모듈, 추가 설치 불필요 |
| HTTP 보안 헤더 조합 | 헤더 직접 나열 | nginx `add_header` 패턴 (D-05 결정) | CSP 등 복잡한 헤더 값 조합 실수 방지 |

**핵심 인사이트:** rate-limiter-flexible은 토큰 버킷 알고리즘의 엣지 케이스(burst handling, overflow, cleanup)를 모두 처리한다. 직접 구현한 Map/counter는 메모리 누수 및 타이머 정리 문제가 발생한다.

## Common Pitfalls

### Pitfall 1: CSP connect-src에 wss:// 미포함

**무엇이 잘못되는가:** `connect-src 'self'`만 설정하면 Socket.IO WebSocket 연결이 브라우저에서 CSP 위반으로 차단된다.
**왜 발생하는가:** `'self'`는 HTTP/HTTPS 스킴만 커버. ws:// 와 wss:// 스킴은 별도로 명시해야 한다 (MDN 공식 문서 확인).
**방지 방법:** `connect-src 'self' wss://sutda.duckdns.org;` 로 wss 도메인 명시.
**경고 신호:** 브라우저 콘솔에 `Refused to connect to 'wss://...' because it violates the Content Security Policy` 오류.

### Pitfall 2: limit_req_zone으로 WebSocket 제한 시도

**무엇이 잘못되는가:** WebSocket 연결 이후의 이벤트 트래픽이 제한되지 않음.
**왜 발생하는가:** `limit_req_zone`은 HTTP 요청당 적용. WebSocket 업그레이드 핸드셰이크에는 적용되지만 이후 프레임 트래픽에는 무효.
**방지 방법:** nginx 레이어는 `limit_conn`으로 동시 연결 수만 제한. 실제 이벤트 빈도 제한은 애플리케이션 레이어(Socket.IO 미들웨어)에서 처리.
**경고 신호:** 이벤트 폭탄 공격 후 nginx 로그에는 정상이지만 Node.js 프로세스 CPU가 100%.

### Pitfall 3: socket.use() 에러가 연결을 끊음

**무엇이 잘못되는가:** `socket.use()` 미들웨어에서 `next(new Error(...))` 호출 시 소켓 연결이 끊어진다. rate limit 시 연결을 유지하면서 해당 이벤트만 무시하려면 다른 패턴이 필요하다.
**왜 발생하는가:** Socket.IO의 `socket.use()` 에러 처리 설계.
**방지 방법:** D-07에서 "이벤트 무시" 결정에 따라 — `next(new Error('RATE_LIMITED'))`로 에러를 발생시키되 클라이언트가 `connect_error` 없이 계속 작동하게 하려면 에러를 catch하고 no-op 처리. 또는 `next()`를 호출하되 핸들러에서 직접 무시하는 패턴 사용.
**경고 신호:** rate limit 발동 후 클라이언트가 연결이 끊어지고 재연결 루프 발생.

### Pitfall 4: nginx add_header가 오류 응답에 미적용

**무엇이 잘못되는가:** `add_header` 지시자는 기본적으로 2xx, 3xx 응답에만 적용된다. 4xx, 5xx 응답에는 헤더가 없어 보안 감사에서 지적될 수 있다.
**방지 방법:** 모든 헤더에 `always` 파라미터 추가: `add_header X-Frame-Options "SAMEORIGIN" always;`

### Pitfall 5: .env가 이미 git에 커밋된 경우

**무엇이 잘못되는가:** `.gitignore`에 추가해도 이미 tracked 파일은 제거되지 않는다.
**방지 방법:** `git status` 확인 후 tracked이면 `git rm --cached .env` 실행. 현재 상태에서 `.env`는 untracked(`??`)으로 확인됨 (git status 확인 완료) — 안전.

## Code Examples

### Rate Limiting 전체 패턴 (TypeScript, packages/server/src/index.ts에 추가)

```typescript
// Source: https://github.com/animir/node-rate-limiter-flexible/wiki/Overall-example (공식 문서)
import { RateLimiterMemory } from 'rate-limiter-flexible';

const socketEventRateLimiter = new RateLimiterMemory({
  points: 20,    // 20개 이벤트
  duration: 1,   // 1초 per window (D-07 결정)
});

io.on('connection', (socket) => {
  // 이벤트별 rate limit (소켓당 초당 20개)
  socket.use(async ([event, ...args], next) => {
    try {
      await socketEventRateLimiter.consume(socket.id);
      next();
    } catch {
      // 한도 초과: 이벤트 무시, 연결 유지
      // next()를 호출하지 않으면 이벤트 핸들러 실행 안 됨
    }
  });

  // ... 기존 이벤트 핸들러
});
```

### nginx 보안 설정 전체 예시 (기존 sutda nginx 설정에 추가)

```nginx
# Source: https://content-security-policy.com/examples/nginx/ + https://nginx.org/en/docs/http/ngx_http_limit_conn_module.html

# http{} 블록 (nginx.conf 또는 conf.d/default.conf):
```

## State of the Art

| 구버전 접근법 | 현재 접근법 | 변경 시점 | 영향 |
|--------------|------------|-----------|------|
| `io.use()` rate limiter (연결당) | `socket.use()` rate limiter (이벤트당) | Socket.IO v4+ | 이벤트 폭탄 방어를 위해 이벤트 레벨 적용 필요 |
| `helmet.js` Node.js 미들웨어 | nginx `add_header` (D-05 결정) | — | 정적 파일 포함 모든 응답에 헤더 적용, Node.js 오버헤드 없음 |
| `limit_req_zone` (요청 rate limit) | `limit_conn` (연결 수 제한) | WebSocket 시대 | WebSocket에는 연결 수 제한이 더 적합 |

**Deprecated/outdated:**
- `socket.io` v2.x CORS 기본 와일드카드 허용: v4에서도 `origin: true` fallback이 여전히 존재 — 명시적 설정 필수
- `rate-limiter-flexible` Redis 백엔드: 단일 서버 + 소규모 게임에서는 오버엔지니어링. `RateLimiterMemory`로 충분.

## Open Questions

1. **Oracle VM `CLIENT_ORIGIN` 환경변수 현재 설정 여부**
   - 알고 있는 것: 코드는 `process.env.CLIENT_ORIGIN || true` — `true`가 fallback
   - 불명확한 것: 실제 VM에 `CLIENT_ORIGIN=https://sutda.duckdns.org` 설정됐는지 SSH 접속 없이 확인 불가
   - 권장: 플랜에서 VM SSH 접속 후 `echo $CLIENT_ORIGIN` 또는 `pm2 env` 확인 태스크 포함

2. **nginx 설정 파일 현재 내용 (sutda 블록)**
   - 알고 있는 것: Phase 12에서 `/etc/nginx/sites-available/sutda`에 설정됨 (12-CONTEXT.md 참조)
   - 불명확한 것: 이미 보안 헤더가 일부 추가됐는지, `location /socket.io/` 블록 존재 여부
   - 권장: 플랜에서 VM에서 `cat /etc/nginx/sites-available/sutda` 실행 후 현재 상태 확인 태스크 포함

3. **socket.use() 미들웨어 에러 처리 동작**
   - 알고 있는 것: `next(error)` 호출 시 클라이언트 `connect_error` 이벤트 발생
   - 불명확한 것: 이벤트를 무시(drop)하면서 연결을 유지하는 공식 패턴 (next를 호출하지 않으면 다음 핸들러 실행 안 됨 — 이 동작이 공식 문서에 명시적으로 보장되지 않음)
   - 권장: `socket.use()`에서 rate limit 초과 시 `next()`를 호출하지 않는 방식 채택. 또는 `try/catch`로 에러를 잡고 rate limit 이벤트를 emit하되 next()를 호출하지 않음.
